# FAQ и Частые ошибки

## 🚨 Частые ошибки студентов

### ❌ Ошибка 1: Слишком мало примеров

**Проблема:**
```
Студент создает промпт с 1-2 примерами для задачи с 5 категориями.
```

**Почему плохо:**
- Модель не видит все категории
- Не понимает различия между категориями
- Низкая точность (<50%)

**Решение:**
```
Минимум 1 пример на категорию
Оптимально: 2-3 примера на категорию

Для 5 категорий:
Минимум: 5 примеров
Оптимально: 10-15 примеров
```

**Пример исправления:**

❌ **ПЛОХО:**
```
Классифицируй: Технический / Оплата / Фича / Жалоба / Общий

Пример:
"Не работает приложение" → Технический
"С карты списали дважды" → Оплата

Классифицируй: "Можно добавить темную тему?"
```

✅ **ХОРОШО:**
```
Классифицируй: Технический / Оплата / Фича / Жалоба / Общий

Примеры:
"Не работает приложение" → Технический
"Ошибка 500 при входе" → Технический
"С карты списали дважды" → Оплата
"Как отменить подписку?" → Оплата
"Добавьте темную тему" → Фича
"Нужна интеграция с Telegram" → Фича
"Поддержка не отвечает третий день!" → Жалоба
"Ужасный сервис" → Жалоба
"Как изменить пароль?" → Общий
"Где инструкция?" → Общий

Классифицируй: "Можно добавить темную тему?"
```

### ❌ Ошибка 2: Все примеры одинаковые

**Проблема:**
```
Все примеры короткие, простые, однообразные
```

**Почему плохо:**
- Модель не видит вариативность
- Не справляется со сложными случаями
- Плохо работает на реальных данных

**Решение:**
```
Примеры должны быть:
✅ Разной длины (короткие и длинные)
✅ Разного стиля (формальные/неформальные)
✅ Разной сложности (простые и сложные)
```

**Пример исправления:**

❌ **ПЛОХО (все короткие и простые):**
```
"Супер!" → позитивный
"Класс!" → позитивный
"Отлично!" → позитивный
```

✅ **ХОРОШО (разнообразные):**
```
"Супер!" → позитивный
"Товар пришел быстро, качество на высоте. Рекомендую!" → позитивный
"Пользуюсь уже месяц, никаких нареканий. За эту цену просто идеально." → позитивный
```


- Ошибки на реальных данных
- Низкая практическая применимость

**Решение:**
```
Включить edge cases:
✅ Отсутствующие данные (null, пустые поля)
✅ Нестандартные форматы
✅ Двусмысленные случаи
✅ Опечатки и сленг
```

**Пример исправления:**

### ❌ Ошибка 4: Непоследовательный формат

**Проблема:**
```
Примеры в разном формате
```

**Почему плохо:**
- Модель не понимает, какой формат использовать
- Непредсказуемый вывод
- Сложно парсить результат

**Решение:**
```
✅ Все примеры в одинаковом формате
✅ Консистентные разделители
✅ Одинаковая структура
```

**Пример исправления:**

❌ **ПЛОХО (разный формат):**
```
Пример 1:
Вход: "текст"
Выход: результат

Пример 2:
"текст" → результат

Пример 3:
Текст: "текст"
Ответ: результат
```

✅ **ХОРОШО (консистентный формат):**
```
Пример 1:
Текст: "текст"
Результат: результат

Пример 2:
Текст: "текст"
Результат: результат

Пример 3:
Текст: "текст"
Результат: результат
```

### ❌ Ошибка 5: Нечеткая инструкция

**Проблема:**
```
"Сделай что-нибудь с этим текстом"
```

**Почему плохо:**
- Модель не понимает, что именно делать
- Результат не соответствует ожиданиям
- Нельзя воспроизвести

**Решение:**
```
✅ Четкая формулировка задачи
✅ Указан формат вывода
✅ Определены категории/структура
✅ Прописаны правила
```

**Пример исправления:**

❌ **ПЛОХО:**
```
Проанализируй отзыв.

"Товар хороший, но дорого" → ?
```

✅ **ХОРОШО:**
```
Определи тон отзыва: позитивный, негативный, нейтральный или смешанный.

Правила:
- Позитивный: в основном хвалят
- Негативный: в основном ругают
- Нейтральный: факты без эмоций
- Смешанный: есть и плюсы, и минусы

"Товар хороший, но дорого" → ?
```

### ❌ Ошибка 6: Не тестируют промпт

**Проблема:**
```
Создали промпт, не протестировали, сдали домашку
```

**Почему плохо:**
- Не знаете реальную точность
- Промпт может не работать
- Пропускаете ошибки

**Решение:**
```
✅ Протестировать минимум на 10 случаях
✅ Включить edge cases
✅ Записать метрики
✅ Итерировать при низкой точности
```

**Процесс:**
```
1. Создать промпт с 3 примерами
2. Протестировать на 10 случаях
3. Точность < 80%?
   → Добавить примеры / улучшить качество
   → Повторить тест
4. Точность > 80%?
   → Готово!
```

### ❌ Ошибка 7: Слишком много примеров

**Проблема:**
```
15 примеров для простой задачи с 2 категориями
```

**Почему плохо:**
- Тратятся токены (деньги)
- Медленнее работает
- Нет улучшения точности

**Решение:**
```
✅ Найти "точку насыщения"
✅ Использовать минимум для нужной точности
✅ Не добавлять примеры "на всякий случай"
```

**Правило:**
```
Начинайте с 3 примеров
Добавляйте по 2-3, если точность низкая
Остановитесь, когда точность перестала расти
```

## ❓ FAQ (Часто задаваемые вопросы)

### Общие вопросы

#### Q: Сколько примеров нужно для Few-shot?

**A:** Зависит от задачи:
- **Простая задача (2-3 категории):** 3-5 примеров
- **Средняя задача (4-6 категорий):** 6-10 примеров
- **Сложная задача (7+ категорий):** 10-20 примеров

**Правило:** Начните с 3, добавляйте, пока точность не достигнет 80-90%.

---

#### Q: Few-shot лучше, чем Zero-shot?

**A:** Не всегда!

**Few-shot лучше, когда:**
- Нужен специфичный стиль/формат
- Нестандартная задача
- Нужна высокая точность

**Zero-shot лучше, когда:**
- Простая задача
- Стандартный формат
- Нужна скорость
- Нет времени на создание примеров

**Практика:** Начните с Zero-shot, переходите к Few-shot, если результат неудовлетворителен.

---

#### Q: Какую модель использовать для Few-shot в 2026?

**A:** Все frontier-модели отлично работают с Few-shot. Выбирайте по задаче:

| Задача | Лучшая модель | Почему |
|--------|---------------|--------|
| Универсально | **GPT-5** ($1.25/M) | Авто-роутинг, structured outputs, 90% кэш |
| Длинный контекст | **Grok 4 Fast** ($0.20/M) | 2M токенов, самый дешёвый |
| Классификация, анализ | **Claude Sonnet 4.6** ($3/M) | Extended thinking, авто-кэш |

| + Веб-поиск | **Perplexity Sonar** | Реальное время, 1200 ток/сек |

**Рекомендации:**
1. **От простого к сложному** — помогает модели
2. **Группировка по категориям** — структурирует
3. **Случайный порядок** — тоже работает

**Эксперимент:** Попробуйте разные порядки и выберите лучший.

### Новые вопросы 2026

#### Q: Что такое Structured Outputs и зачем они нужны?

**A:** Structured Outputs — это гарантия формата ответа (JSON Schema). Все ведущие модели (GPT-5, Claude 4.6, Gemini 3.1 Pro) поддерживают их.

**Как это меняет Few-shot:**
```
РАНЬШЕ: 5 примеров = 3 на формат + 2 на содержание
СЕЙЧАС: Structured Outputs = формат, 5 примеров = ВСЕ на содержание
```

**Практический совет:** Используйте Structured Outputs для JSON/HTML/SQL формата, а few-shot примеры — для обучения стилю, тону и логике.

#### Q: Что такое Prompt Caching и как это экономит деньги?

**A:** Prompt Caching кэширует статические части промпта (системную инструкцию + примеры). При повторном использовании вы платите в 5-20 раз меньше.

```
Пример: промпт с 20 примерами (5000 токенов)
Первый запрос:  $0.015 (полная цена)
Следующие:      $0.0015 (кэш, 90% скидка)

1000 запросов: $1.50 вместо $15.00
```

**Как включить:**
- **GPT-5, Gemini, DeepSeek:** автоматически (ничего не нужно делать)
- **Claude:** автоматически с февраля 2026

**Совет:** Размещайте few-shot примеры в начале промпта, переменные данные — в конце.

#### Q: Как измерить качество Few-shot промпта?

**A:** Метрики:

**1. Точность (Accuracy):**
```
Точность = Правильных ответов / Всего тестов × 100%
Цель: > 80-90%
```

**2. Precision & Recall (для классификации):**
```
Precision = True Positives / (True Positives + False Positives)
Recall = True Positives / (True Positives + False Negatives)
```

**3. Консистентность:**
```
Запустить промпт 5 раз на одних данных
Консистентность = Одинаковых результатов / 5
```

**4. Скорость:**
```
Среднее время на 1 запрос
```

**5. Стоимость:**
```
Токены на 1 запрос × Цена за токен
```

## 🔧 Troubleshooting Guide

### Проблема: Точность < 50%

**Диагностика:**
1. [ ] Достаточно ли примеров?
2. [ ] Примеры разнообразные?
3. [ ] Инструкция четкая?
4. [ ] Правильные метки?

**Решения:**
- ✅ Добавить больше примеров (5 → 8 → 10)
- ✅ Улучшить разнообразие примеров
- ✅ Уточнить инструкцию
- ✅ Проверить метки

---

### Проблема: Модель не следует формату

**Диагностика:**
1. [ ] Примеры в одинаковом формате?
2. [ ] Формат четко показан?
3. [ ] Есть инструкция о формате?

**Решения:**
- ✅ Унифицировать формат примеров
- ✅ Добавить: "Строго соблюдай формат"
- ✅ Показать больше примеров формата
- ✅ Использовать Structured Outputs (JSON Schema в GPT-5, Claude 4.6, Gemini 3.1 Pro)

---

### Проблема: Модель путает категории A и B

**Диагностика:**
1. [ ] Есть ли примеры обеих категорий?
2. [ ] Четко ли различие между A и B?
3. [ ] Есть ли граничные примеры?

**Решения:**
- ✅ Добавить больше примеров A и B
- ✅ Добавить пример "A, похожий на B"
- ✅ Добавить пример "B, похожий на A"
- ✅ Добавить правило различения:
```
A — если [условие]
B — если [условие]
```

---

### Проблема: Слишком дорого (много токенов)

**Диагностика:**
1. [ ] Сколько примеров используется?
2. [ ] Насколько длинные примеры?
3. [ ] Есть ли "точка насыщения"?

**Решения:**
- ✅ Найти минимум примеров для 80% точности
- ✅ Сократить длинные примеры (убрать лишнее)
- ✅ Использовать более дешевую модель
- ✅ Кэшировать результаты

---

### Проблема: Медленно работает

**Решения:**
- ✅ Уменьшить количество примеров
- ✅ Использовать более быструю модель (Grok 4 Fast или DeepSeek V3 вместо GPT-5 Pro)
- ✅ Батчить запросы
- ✅ Использовать асинхронные вызовы

## 💡 Лучшие практики

### ✅ DO (Делайте так)

1. **Начинайте с 3 примеров**
   ```
   3 примера → тестируйте → добавляйте при необходимости
   ```

2. **Тестируйте на реальных данных**
   ```
   Минимум 10 тестовых случаев с реальными данными
   ```

3. **Включайте edge cases**
   ```
   Missing values, опечатки, нестандартные форматы
   ```

4. **Итерируйте**
   ```
   V1 → тест → улучшение → V2 → тест → V3 ...
   ```

5. **Сохраняйте успешные промпты**
   ```
   Создавайте библиотеку для переиспользования
   ```

### ❌ DON'T (Не делайте так)

1. **Не используйте слишком мало примеров**
   ```
   ❌ 1-2 примера для сложной задачи
   ```

2. **Не делайте все примеры одинаковыми**
   ```
   ❌ Только короткие и простые примеры
   ```

3. **Не пропускайте тестирование**
   ```
   ❌ Создали промпт → не протестировали → сдали
   ```

4. **Не игнорируйте edge cases**
   ```
   ❌ Только "идеальные" примеры
   ```

5. **Не используйте плохие примеры**
   ```
   ❌ Примеры с ошибками или неправильными метками
   ```

## 📊 Чек-лист качества промпта

Используйте перед сдачей домашки:

### Структура промпта:
- [ ] Четкая инструкция
- [ ] Минимум 3 примера
- [ ] Примеры в едином формате
- [ ] Указан формат вывода

### Качество примеров:
- [ ] Разнообразные (длина, стиль, сложность)
- [ ] Репрезентативные (похожи на реальные)
- [ ] Без ошибок
- [ ] Включают edge cases

### Тестирование:
- [ ] Протестировано на минимум 10 случаях
- [ ] Включены edge cases в тестирование
- [ ] Точность > 70%
- [ ] Записаны метрики

### Оптимизация:
- [ ] Найдено оптимальное количество примеров
- [ ] Убраны избыточные примеры
- [ ] Промпт работает стабильно

### Документация:
- [ ] Промпт сохранен
- [ ] Есть инструкция по использованию
- [ ] Записаны результаты тестирования

## 📞 Когда обращаться к преподавателю

Обращайтесь, если:

1. **Низкая точность после 3+ итераций**
   > "Попробовал 5 раз улучшить промпт, точность все еще 60%"

2. **Не понимаете, почему модель ошибается**
   > "Модель классифицирует все как категорию A"

3. **Задача кажется слишком сложной для Few-shot**
   > "Не уверен, подходит ли Few-shot для моей задачи"

4. **Технические проблемы**
   > "API возвращает ошибку 500"

5. **Нужна помощь с выбором задачи**
   > "Не могу придумать задачу для домашнего задания"

**НЕ бойтесь спрашивать!** Преподаватель здесь, чтобы помочь.
