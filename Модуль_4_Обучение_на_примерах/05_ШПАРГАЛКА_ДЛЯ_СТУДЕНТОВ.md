# Шпаргалка: Few-shot Prompting

## 🎯 Основные концепции

### Zero-shot vs Few-shot vs Many-shot

```
┌─────────────────┬──────────────┬────────────────┬──────────────┐
│     Метод       │   Примеры    │   Точность     │  Скорость    │
├─────────────────┼──────────────┼────────────────┼──────────────┤
│ Zero-shot       │      0       │     ★★☆☆☆      │    ★★★★★     │
│ Few-shot        │     3-10     │     ★★★★☆      │    ★★★☆☆     │
│ Many-shot       │    50-1000   │     ★★★★★      │    ★☆☆☆☆     │
└─────────────────┴──────────────┴────────────────┴──────────────┘
```

**Когда использовать:**
- **Zero-shot:** Простые задачи, стандартные форматы
- **Few-shot:** Специфичный стиль, нестандартный формат
- **Many-shot:** Максимальная точность, сложные задачи

## 🎨 Примеры для разных задач

### 1. Классификация

```
Классифицируй отзыв: позитивный, негативный или нейтральный.

Примеры:
"Отлично! Рекомендую!" → позитивный
"Ужасное качество." → негативный
"Обычный товар." → нейтральный

Классифицируй: "[ВАШ ТЕКСТ]"
```

**Правила:**
- Минимум 2 примера на категорию
- Примеры разной длины и стиля
- Покрывают граничные случаи

---

### 2. Извлечение данных

```
Извлеки email, телефон и имя из текста в JSON.

Примеры:

Текст: "Иван Петров, ivan@mail.ru, +375291234567"
JSON: {
  "имя": "Иван Петров",
  "email": "ivan@mail.ru",
  "телефон": "+375291234567"
}

Текст: "Контакт: anna@company.com"
JSON: {
  "имя": null,
  "email": "anna@company.com",
  "телефон": null
}

---

Извлеки данные:
Текст: "[ВАШ ТЕКСТ]"
JSON:
```

**Правила:**
- Показать обработку missing values (null)
- Разные форматы входных данных
- Строгий формат вывода

---

### 3. Генерация текста

```
Напиши описание товара для маркетплейса (продающее, конкретное, 2-3 предложения).

Примеры:

Товар: Умная колонка Яндекс Станция
Описание: Ваш личный помощник в компактном корпусе. Включает музыку голосом, управляет умным домом. Размером с кофейную чашку — поместится везде.

Товар: Механическая клавиатура Keychron K2
Описание: Почувствуйте разницу при каждом нажатии. Переключатели Gateron Brown обеспечивают тактильный отклик без шума. Работает по Bluetooth с 3 устройствами одновременно.

---

Товар: [ВАШ ТОВАР]
Описание:
```

**Правила:**
- Показать стиль и тон
- Демонстрировать структуру
- Примеры разной длины

---

### 4. Форматирование

```
Преобразуй описание в HTML-карточку.

Примеры:

Описание: MacBook Pro 14, M3, 18GB RAM, Space Gray
HTML:
<div class="product">
  <h3>MacBook Pro 14"</h3>
  <p>M3 • 18GB RAM</p>
  <span>Space Gray</span>
</div>

Описание: iPhone 15 Pro, 256GB, Titanium Blue
HTML:
<div class="product">
  <h3>iPhone 15 Pro</h3>
  <p>256GB</p>
  <span>Titanium Blue</span>
</div>

---

Описание: [ВАШЕ ОПИСАНИЕ]
HTML:
```

**Правила:**
- Строгое соблюдение формата
- Показать все элементы структуры
- Консистентность между примерами

## 💡 Правила подбора примеров

### ✅ Хорошие примеры

1. **Разнообразные**
   - Разная длина (короткие и длинные)
   - Разный стиль (формальный/неформальный)
   - Разные edge cases

2. **Репрезентативные**
   - Похожи на реальные данные
   - Покрывают типичные случаи
   - Включают сложные случаи

3. **Качественные**
   - Без ошибок
   - С правильными метками
   - Понятные и четкие

4. **Краткие**
   - Не перегружают контекст
   - Фокус на главном
   - Убраны лишние детали

### ❌ Плохие примеры

1. **Однообразные**
   - Все короткие/простые
   - Одинаковый стиль
   - Не покрывают вариативность

2. **Нерепрезентативные**
   - Слишком простые
   - Не похожи на реальность
   - Не покрывают edge cases

3. **Некачественные**
   - С ошибками
   - С неправильными метками
   - Противоречивые

4. **Избыточные**
   - Слишком длинные
   - Лишние детали
   - Тратят контекст

## 📏 Оптимальное количество примеров

### Базовые правила

```
Простая задача (2-3 категории):     3-5 примеров
Средняя задача (4-6 категорий):     6-10 примеров
Сложная задача (7+ категорий):      10-20 примеров
Очень сложная (много edge cases):   20-50 примеров
```

### Процесс оптимизации

```
1. Начни с 3 примеров
        ↓
2. Протестируй на 10 случаях
        ↓
3. Точность < 80%?
    Да → Добавь 2-3 примера, повтори
    Нет → Готово!
        ↓
4. Найди минимум, при котором точность > 90%
```

### Правило "точки насыщения"

> После определенного количества примеров точность перестает расти. Найди эту точку и не трать лишние токены.

**Пример:**
```
3 примера  → 60% точность
5 примеров  → 80% точность
8 примеров  → 92% точность
10 примеров → 93% точность  ← Точка насыщения
15 примеров → 93% точность  ← Лишние примеры
```

**Вывод:** Оптимум — 8-10 примеров

## 🔧 Troubleshooting

### Проблема: Модель не следует формату

**Решения:**
1. ✅ Добавить больше примеров (5 → 8)
2. ✅ Усилить инструкцию: "Строго соблюдай формат"
3. ✅ Проверить консистентность примеров
4. ✅ Добавить пример с edge case

**Пример:**
```
❌ ПЛОХО:
Примеры в разном формате

✅ ХОРОШО:
Все примеры в одинаковом формате + "Строго соблюдай формат JSON"
```

---

### Проблема: Модель путает две категории

**Решения:**
1. ✅ Добавить примеры граничных случаев
2. ✅ Добавить пример A, похожий на B
3. ✅ Добавить пример B, похожий на A
4. ✅ Уточнить инструкцию с правилами различения

**Пример:**
```
Модель путает "важно" и "срочно"
        ↓
Добавить пример: "Важно, но не срочно: согласовать договор до пятницы"
Добавить пример: "Срочно, но не критично: ответить клиенту сегодня"
```

---

### Проблема: Низкая точность на edge cases

**Решения:**
1. ✅ Включить edge cases в примеры
2. ✅ Добавить правило обработки edge cases
3. ✅ Увеличить количество примеров
4. ✅ Попробовать другую модель

**Пример:**
```
❌ ПЛОХО (только простые случаи):
"Иван Петров, ivan@mail.ru" → {"имя": "Иван Петров", "email": "ivan@mail.ru"}

✅ ХОРОШО (с edge cases):
"Иван Петров, ivan@mail.ru" → {"имя": "Иван Петров", "email": "ivan@mail.ru"}
"Контакт: anna@mail.ru" → {"имя": null, "email": "anna@mail.ru"}
"Дмитрий" → {"имя": "Дмитрий", "email": null}
```

---

### Проблема: Слишком много токенов

**Решения:**
1. ✅ Уменьшить количество примеров
2. ✅ Сократить примеры (убрать лишнее)
3. ✅ Использовать Many-shot только для сложных задач
4. ✅ Найти "точку насыщения"

**Пример:**
```
❌ ПЛОХО (избыточный пример):
Текст: "Это длинный текст с множеством деталей, которые не влияют на классификацию, но занимают место..."
Категория: позитивный

✅ ХОРОШО (краткий):
Текст: "Отличный товар!"
Категория: позитивный
```

## 🎓 Лайфхаки

### 1. Генерация примеров через AI

Вместо ручного создания примеров:

```
Сгенерируй 5 примеров email с категорией "Срочно" для обучения классификатора:
- Разной длины
- Разного стиля
- С разной степенью срочности
```

---

### 2. Использование реальных данных

```
1. Возьми 20 реальных примеров из работы
2. Разметь их вручную
3. Выбери 5-8 самых разнообразных
4. Используй в промпте
```

---

### 3. Итеративное улучшение

```
Версия 1: 3 примера → 60% точность
    ↓ (добавить примеры для ошибочных случаев)
Версия 2: 5 примеров → 80% точность
    ↓ (добавить edge cases)
Версия 3: 8 примеров → 92% точность
    ↓ (оптимизация)
Финал: 6 примеров → 90% точность (убрали избыточные)
```

---

### 4. A/B тестирование примеров

```
Создай 2 набора примеров:
- Набор A: [примеры 1-5]
- Набор B: [примеры 6-10]

Протестируй оба на одинаковых данных
Выбери лучший или комбинируй
```

---

### 5. Порядок примеров имеет значение

```
От простого к сложному:
Пример 1: Простой случай
Пример 2: Средний случай
Пример 3: Сложный случай

Или группировка по категориям:
Примеры категории A
Примеры категории B
Примеры категории C
```

## 📋 Чек-лист создания Few-shot промпта

### Перед созданием:
- [ ] Четко определена задача
- [ ] Понятны входные и выходные данные
- [ ] Определены категории (для классификации)
- [ ] Собраны реальные примеры для тестирования

### Создание промпта:
- [ ] Написана четкая инструкция
- [ ] Добавлено минимум 3 примера
- [ ] Примеры разнообразные
- [ ] Примеры качественные (без ошибок)
- [ ] Формат вывода четко показан
- [ ] Добавлены правила (если нужно)

### Тестирование:
- [ ] Протестировано на минимум 10 случаях
- [ ] Включены edge cases в тестирование
- [ ] Точность > 70%
- [ ] Записаны метрики

### Оптимизация:
- [ ] Найдена "точка насыщения"
- [ ] Убраны избыточные примеры
- [ ] Промпт работает стабильно
- [ ] Промпт сохранен для переиспользования

## 🆕 Новые техники 2026 года

### 1. Structured Outputs (Constrained Decoding)

```
РАНЬШЕ (2024): Few-shot примеры = формат + стиль
СЕЙЧАС (2026): Structured Outputs = формат (гарантированно)
               Few-shot примеры = ТОЛЬКО стиль и логика

Поддерживают: GPT-5, Claude 4.6, Gemini 3.1 Pro
```

**Что это значит:** Больше не нужно тратить примеры на обучение формату JSON/HTML/SQL. Указываете JSON Schema — модель гарантированно вернёт валидный JSON. Примеры теперь только для стиля и содержания.

---

### 2. Prompt Caching

```
┌────────────────────┬──────────────┬──────────────┐
│ Провайдер          │ Скидка       │ Тип          │
├────────────────────┼──────────────┼──────────────┤
│ GPT-5              │ до 90%       │ авто         │
│ Claude 4.6         │ до 90%       │ авто (2026)  │
│ DeepSeek V4        │ 75-90%       │ авто         │
│ Gemini 3.1         │ значит.      │ авто         │
│ Grok 4 Fast        │ ~75%         │ по запросу   │
└────────────────────┴──────────────┴──────────────┘

Совет: размещайте few-shot примеры в начале промпта
(системная инструкция), а переменные данные — в конце.
Так примеры кэшируются, и вы платите за них только 1 раз.
```

### 3. Мультимодальный Few-shot

```
Текстовый few-shot:
  "Описание → Категория" (текст → текст)

Мультимодальный few-shot (2026):
  [Фото товара] → "Дефект: царапина" (изображение → текст)
  [Скриншот UI] → "Проблема: кнопка обрезана" (изображение → текст)
  [YouTube URL] → "Резюме: обзор продукта" (видео → текст)

Лучшая модель: Gemini 3.1 Pro (до 900 изображений, аудио, видео)
Также: Claude 4.6, GPT-5 (изображения)
```

---

### 4. Chain-of-Thought Few-shot для Reasoning-моделей

```
Обычный few-shot:
  Вход → Ответ

Chain-of-Thought few-shot (для GPT-5 Thinking, Claude Extended Thinking):
  Вход → Рассуждение → Ответ

Пример:
  Задача: "15% от 240"
  Рассуждение: "15% = 0.15. 240 × 0.15 = 36"
  Ответ: 36

Совет: для reasoning-моделей показывайте ХОД рассуждений,
не только вход-выход. Это повышает точность на 10-50%.
```

---

### 5. Ключевой сдвиг: Качество > Количество

```
Модели 2024:  10-20 примеров для сложных задач
Модели 2026:  3-5 качественных примеров часто достаточно

Почему? Frontier-модели (GPT-5, Claude 4.6, Grok 4)
лучше обобщают из малого числа примеров.

Один хорошо подобранный пример > десять посредственных
```

## 🎯 Финальные советы

### 1. Начинай с простого
> Не усложняй. 3 примера лучше, чем 0. Всегда можно добавить больше.

### 2. Тестируй на реальных данных
> Промпт, который работает на игрушечных примерах, может не работать в продакшене.

### 3. Итерируй
> Первая версия промпта редко идеальна. Улучшай постепенно.

### 4. Сохраняй успешные промпты
> Создавай библиотеку. Это инвестиция в будущую продуктивность.

### 5. Делись с командой
> Хороший промпт экономит время всем. Создавайте командную базу знаний.

## 🔥 Quick Reference

### Минимальный рабочий промпт

```
Задача: [что делать]

Примеры:
[пример 1]
[пример 2]
[пример 3]

---

[ваши данные]
```

### Правило 3-10-90

- **3** примера — минимум
- **10** тестовых случаев — минимум
- **90%** точность — цель

### Формула оценки

```
Точность = Правильных ответов / Всего тестов × 100%

Оптимальность = Точность / Количество примеров

Цель: Максимальная точность при минимуме примеров
```
